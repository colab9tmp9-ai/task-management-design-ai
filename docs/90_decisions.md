# 決定事項

このファイルには、採用済みの設計判断だけを書く。

AIの提案や議論の途中経過は `99_ai_discussion_log.md` に書く。迷っていることは `91_open_questions.md` に書く。

## 2026-04-26

### GitHub運用

- 設計ログ置き場として、publicリポジトリを使う。
- リポジトリ名は `task-management-design-ai` とする。
- GitHubには公開してよい設計情報だけを書く。
- 実際の予定、タスク、企業名、学校名、面接日時、メール本文、Slack本文などはGitHubに書かない。
- ChatGPTとClaudeには、このリポジトリのファイルを参照させる。

### 情報の置き場所

- 設計ドキュメントはGitHubに置く。
- 実際のイベントはGoogleカレンダーに置く候補とする。
- 実際のタスクはNotionに置く候補とする。
- 個人情報や実データは、GitHubではない非公開の場所に置く。

### AIの役割

- ChatGPTとClaudeは、設計の相談、比較、レビューに使う。
- 両AIの回答は、全文ではなく要約して記録する。
- 採用する内容は人間が判断する。
- 採用済みの内容だけをこのファイルに移す。

## 2026-04-27 - 初期設計レビュー後の決定

### 基本方針

- イベントの正本はGoogleカレンダーとする。
- タスクの正本はNotion候補とする。
- 案件は、複数のタスクやイベントをまとめる上位概念として導入する方針とする。
- 設計文書の正本はGitHubとする。
- GitHubには、設計情報のみを置く。
- GitHubには、実予定、実タスク、個人情報、仮称対応表を置かない。

### ツールの役割分担

- Googleカレンダーは、日時が決まっているイベントを管理する。
- Notionは、タスク、案件、確認待ち候補を管理する候補とする。
- GitHubは、要件、分類ルール、決定事項、未決定事項、AI相談ログを管理する。
- Pythonは、日常的な自動化に使う候補とする。
- ChatGPTとClaudeは、設計レビュー、優先順位相談、文章作成補助に使う。

### 自動化方針

- Pythonによる最初の自動化は、完全自動登録ではなく、確認待ち候補の作成や朝サマリー生成を中心にする。
- メール、チャット、カレンダー等から抽出した情報は、いきなり正式タスクとして登録しない。
- 自動抽出した情報は、まず確認待ちとして扱う方針とする。
- AIは毎日の実行処理の中心にはしない。
- AIは週1回程度の設計レビュー、優先順位相談、文章作成補助に使う方針とする。

### 次の設計対象

- 次に優先して決める対象は、イベント・タスク・案件の分類ルールとする。
- 分類ルールを固めた後に、Notionの最小DB構成を決める。
- Notion DB設計では、最初から複雑な構成にせず、小さく始めることを重視する。
