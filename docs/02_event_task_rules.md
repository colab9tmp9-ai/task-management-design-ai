# イベント・タスク・案件の分類ルール

## 目的

このファイルでは、スケジュール・タスク管理システムにおいて、情報をどのように分類するかを定義する。

分類ルールを明確にすることで、GoogleカレンダーとNotionに同じ情報を重複して登録したり、重要な作業を見落としたりすることを防ぐ。

## 基本分類

管理対象は、主に以下の3種類に分ける。

- イベント
- タスク
- 案件

## イベント

イベントとは、特定の日時に実施される予定である。

例：

- 授業
- 面接
- ミーティング
- アルバイトの勤務
- 研究室の予定
- 締め切り当日の予定化された作業時間

保存先の正本は Googleカレンダー とする。

### イベントとして登録する条件

次のいずれかに当てはまる場合は、イベントとして扱う。

- 開始時刻と終了時刻が決まっている
- その時間に参加、出席、移動、対応する必要がある
- 通知を受け取らないと忘れる危険がある
- オンライン会議URLや場所が必要である

### イベントに含める情報

- 日時
- 場所、URL、住所
- オンライン / オフライン
- 持ち物
- 服装
- 事前準備
- 関連タスク
- 関係者や連絡先

## タスク

タスクとは、期限までに完了すべき作業や手続きである。

例：

- メールに返信する
- フォームを提出する
- レポートを書く
- 面接準備をする
- 教材を進める
- 研究資料を読む
- Slackの連絡に対応する

保存先の正本は Notion とする候補である。

### タスクとして登録する条件

次のいずれかに当てはまる場合は、タスクとして扱う。

- 完了 / 未完了で状態を管理できる
- 締め切りがある
- 提出先や提出方法がある
- 後で対応しないといけない
- 何らかの準備や作業が必要である

### タスクに含める情報

- タイトル
- カテゴリ
- 締め切り
- 優先度
- ステータス
- 提出先、提出方法
- 情報源
- 関連案件
- メモ

## 案件

案件とは、複数のイベントやタスクをまとめる単位である。

例：

- 授業A
- 研究テーマA
- 企業Aの選考
- インターンA
- アルバイトA
- 個人プロジェクトA

案件は、Notionで管理する候補である。

### 案件として登録する条件

次のいずれかに当てはまる場合は、案件として扱う。

- 複数のタスクが発生する
- 複数のイベントが発生する
- 継続的に管理する必要がある
- 進捗や状況をまとめて確認したい
- 連絡履歴や関連メモを残したい

## 迷ったときの判断ルール

### 日時が決まっている場合

日時が決まっていて、その時間に行動する必要があるならイベントにする。

例：

- 面接
- 授業
- ミーティング
- 勤務

### 期限までに終わればよい場合

特定の時刻に行動する必要はなく、期限までに完了すればよいならタスクにする。

例：

- レポート提出
- メール返信
- 資料作成
- フォーム回答

### 継続的に関連情報が増える場合

複数の予定や作業が紐づくなら案件にする。

例：

- 企業Aの選考
- 授業A
- 研究テーマA

## 重複登録の方針

- イベントの正本はGoogleカレンダー
- タスクの正本はNotion
- 案件の正本はNotion
- Googleカレンダーには、タスクの詳細を書きすぎない
- Notionには、必要に応じて関連イベントへのリンクを置く
- 同じ情報を複数箇所に手入力しないようにする
